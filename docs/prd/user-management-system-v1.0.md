# PRD - 用户管理系统 v1.0

## 1. 需求概述

### 1.1 需求来源
用户提出"设计一个简单的用户管理系统"需求。

### 1.2 需求目标
构建一套轻量、易用的用户管理后台，支持用户的增删改查、权限分配、登录认证等核心能力，满足日常业务运营需求。

### 1.3 目标用户
- 系统管理员：负责用户账号管理和权限分配
- 普通用户：注册、登录、个人信息管理

---

## 2. 功能需求

### 2.1 用户认证模块

#### 2.1.1 登录
- 支持手机号 + 短信验证码登录（默认方案）
- 支持账号密码登录（可选）
- 登录失败锁定：连续5次失败锁定账号30分钟
- 支持"记住登录状态"（7天免登录）

#### 2.1.2 注册
- 手机号注册（验证码校验）
- 用户名、密码设置
- 注册即默认分配"普通用户"角色

#### 2.1.3 找回密码
- 手机号验证码重置密码
- 重置后旧密码立即失效

#### 2.1.4 退出登录
- 清除登录态（本地 + 服务端）

### 2.2 用户管理模块（管理员功能）

#### 2.2.1 用户列表
- 分页展示（默认20条/页，支持10/20/50条切换）
- 支持按用户名/手机号模糊搜索
- 支持按角色、状态（启用/禁用）筛选
- 显示字段：ID、用户名、手机号、角色、状态、注册时间、最后登录时间

#### 2.2.2 新增用户
- 管理员手动创建账号（用户名、手机号、初始密码、角色分配）
- 创建后自动发送短信通知

#### 2.2.3 编辑用户
- 修改用户名、角色、状态
- 手机号修改需验证码确认

#### 2.2.4 禁用/启用用户
- 禁用后该用户无法登录
- 已在线用户强制下线

#### 2.2.5 删除用户
- 软删除（标记deleted_at，不物理删除）
- 删除后数据保留90天可恢复

#### 2.2.6 重置密码
- 管理员可重置任意用户密码
- 重置后用户首次登录强制修改密码

### 2.3 角色与权限模块

#### 2.3.1 预设角色
| 角色       | 权限说明                     |
| ---------- | ---------------------------- |
| 超级管理员 | 全部权限，含角色管理         |
| 管理员     | 用户管理、数据查看           |
| 普通用户   | 仅个人信息管理               |

#### 2.3.2 权限分配
- 管理员可为用户分配/修改角色
- 一个用户仅可拥有一个角色（v1.0简化设计）

### 2.4 个人信息模块

#### 2.4.1 查看/编辑个人信息
- 头像上传（支持jpg/png，最大2MB）
- 昵称、手机号、邮箱（可选）
- 修改密码（需验证旧密码）

#### 2.4.2 登录记录
- 展示最近10条登录记录（时间、IP、设备）

### 2.5 操作日志（管理员可见）

#### 2.5.1 日志记录
- 记录关键操作：登录、创建用户、修改角色、禁用/启用、删除
- 记录字段：操作人、操作时间、操作类型、操作对象、IP

#### 2.5.2 日志查询
- 支持按操作人、操作类型、时间范围筛选

## 3. 非功能性需求

| 类别         | 要求                    | 说明                          |
| ------------ | ----------------------- | ----------------------------- |
| 性能         | 页面加载 < 2秒          | 用户列表页首屏渲染时间        |
| 并发         | 1000 QPS 以内           | 满足中小企业规模              |
| 数据安全     | 密码 bcrypt 加密存储    | 禁止明文存储                  |
| 传输安全     | 全站 HTTPS              | 防止中间人攻击                |
| 数据保留     | 操作日志保留90天        | 软删除用户数据保留90天可恢复  |
| 兼容性       | 主流浏览器 + 移动端适配 | Chrome/Edge/Safari/Firefox    |
| 短信服务     | 阿里云/腾讯云短信       | 验证码有效期5分钟             |

## 4. 边界条件与异常处理

| 场景                   | 处理方式                                  |
| ---------------------- | ----------------------------------------- |
| 手机号已被注册         | 提示"该手机号已注册"，引导登录或找回密码  |
| 验证码过期             | 提示"验证码已过期，请重新获取"            |
| 验证码错误次数超限     | 连续3次错误，锁定验证码功能10分钟         |
| 并发编辑同一用户       | 乐观锁机制，后提交者提示"数据已被修改"    |
| 短信服务异常           | 降级为邮件验证码（如已配置邮箱）          |
| 用户被禁用后仍有会话   | 服务端校验token时拦截，返回401并强制登出  |

## 5. 假设清单

| 编号 | 假设内容                                     | 依据                     | 风险等级 |
| ---- | -------------------------------------------- | ------------------------ | -------- |
| A01  | 用户规模在10000以内                          | "简单"系统定位           | 低       |
| A02  | 不需要多级组织架构/部门管理                  | 未提及，v1.0保持精简     | 中       |
| A03  | 不需要OAuth第三方登录（微信/钉钉等）         | 未提及，后续可迭代       | 低       |
| A04  | 短信验证码为主要认证方式                     | 行业默认方案             | 低       |
| A05  | 角色模型采用RBAC（基于角色的访问控制）       | 最通用的权限方案         | 低       |
| A06  | 单用户单角色，不支持多角色叠加               | 简化设计，降低复杂度     | 中       |
| A07  | 无审计合规要求（如等保三级）                 | 未提及                   | 中       |
| A08  | 前端采用响应式设计，PC/移动端一套代码        | 降低开发成本             | 低       |
